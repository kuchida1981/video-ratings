## Context

作品一覧テーブル表示（`WorkTable` コンポーネント）は現在すべて読み取り専用。カスタムフィールドの編集には作品詳細ページへの遷移が必要。カスタムフィールドの更新API（`PATCH /works/{id}/custom-fields`）と対応するクライアント関数（`api.works.updateCustomFields`）は既に存在しており、バックエンド変更は不要。

関連する既存コンポーネント:
- `WorkTable`（`frontend/src/components/WorkTable.tsx`）: テーブル描画。props で `works`, `visibleColumns`, `customFieldDefs` を受け取る。
- `WorksPage`（`frontend/src/pages/WorksPage.tsx`）: viewMode 管理、列トグル、フィルタ・ソート。

## Goals / Non-Goals

**Goals:**
- テーブル表示でカスタムフィールドをインライン編集できるようにする
- ページ遷移なしで複数作品のカスタムフィールドを連続編集できるようにする
- 編集モードのON/OFFを切り替えられるようにする

**Non-Goals:**
- タイトル・出演者・スコア・タグ・ファイル数・登録日の編集
- タイル表示での編集
- バッチ更新API（一括送信）の追加。セル単位で既存APIを呼ぶ。
- 出演者一覧テーブルの編集モード（将来の拡張候補）

## Decisions

### 1. 編集モード状態の管理場所

**決定**: `WorksPage` で `editMode: boolean` 状態を管理し、`WorkTable` に props として渡す。

**理由**: 編集モードのトグルUIはテーブル上部のヘッダー領域（viewMode トグルの隣）に置くのが自然。`WorksPage` が viewMode を管理しているため、editMode も同じレベルで管理する。editMode は localStorage に保存しない（ページ遷移でリセット）。

### 2. セル入力UIの実装方式

**決定**: 編集モードON時に、カスタムフィールド列の `<td>` 内を型に応じた `<input>` に置き換える。

| フィールド型 | 入力UI |
|---|---|
| text | `<input type="text">` |
| number | `<input type="number">` |
| date | `<input type="date">` |
| boolean | `<input type="checkbox">` |

**代替案**: セルダブルクリックで個別に編集モードに入る方式も検討したが、「編集モード」として一括切り替えするほうが、操作がわかりやすくTab移動で連続入力しやすい。

### 3. 保存タイミングとAPI呼び出し

**決定**: blur イベント時に変更を検出し、変更があった場合のみ `api.works.updateCustomFields(workId, { [fieldName]: value })` を呼ぶ。boolean 型は change イベントで即保存する（チェックボックスは blur のタイミングが不自然なため）。

**変更検出**: 編集モードON時の初期値をローカルに保持し、blur 時に比較する。値が同じなら API 呼び出しをスキップする。

**楽観的更新**: API 呼び出し前にローカル状態を更新し、失敗時にロールバックする。TanStack Query の `invalidateQueries` でリスト全体のリフェッチは行わず、ローカル状態で管理する。ただし編集モードを終了する際にリストを再取得して最新状態と同期する。

### 4. 保存フィードバック

**決定**: 最小限のフィードバック。成功時はサイレント（UI変化なし）、失敗時のみセル枠を赤くハイライトしてエラーを伝える。

**理由**: 連続編集の流れを妨げないことを優先する。保存中のローディング表示も省略する（API は軽量で高速なため）。

### 5. 編集モード中の行クリック遷移の扱い

**決定**: 編集モード中はタイトル列のリンクは維持するが、カスタムフィールドセルのクリックは入力フォーカスに使われるため、行全体のクリック遷移は無効化する。

**理由**: 編集中にうっかり行クリックでページ遷移すると編集内容が失われる可能性があり、UXが悪化する。タイトル列のリンクは残すので、詳細ページへの導線は維持される。

### 6. カスタムフィールド列が非表示の場合

**決定**: 編集モードは表示中のカスタムフィールド列のみ対象とする。列が非表示なら編集UIも表示しない。カスタムフィールド列が1つも表示されていない場合、編集モードボタン自体を無効化（disabled）する。

## Risks / Trade-offs

- **[大量行の編集]** → 一覧に数百行あるとinput要素が大量に描画される。ただし現状ページネーションがないため、既に全行を描画している。input への変換で追加されるDOM量は限定的で、パフォーマンス問題は起きにくい。
- **[同時編集の競合]** → シングルユーザー想定のアプリなので考慮不要。
- **[blur保存の操作感]** → Tab キーで次のセルに移ると blur → 保存が走る。API が遅い場合にユーザーが気づかず次のセルを編集する可能性がある。→ ミティゲーション: 失敗時にセル枠を赤くして気づかせる。
