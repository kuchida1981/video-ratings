## Context

作品一覧および出演者一覧（タイル表示・テーブル表示）において、各要素のクリックハンドリングを JavaScript（`useNavigate`）で行っているため、ブラウザ標準のリンクショートカット（Shift+中クリックなど）が機能しない。

## Goals / Non-Goals

**Goals:**
- 各リストのアイテムを `Link` コンポーネント化し、ブラウザの標準的な「新しいタブで開く」ショートカットが動作するようにする。
- 既存のタイルやテーブルの見た目（CSS スタイル）を崩さずに実装する。

**Non-Goals:**
- 詳細画面（`WorkDetailPage` や `PerformerDetailPage`）自体の内部遷移やデザインの変更。
- テーブルレイアウトの完全な Grid 化（今回はテーブルタグ構造を維持しつつ、特定のセルにリンクを配置する形で対応する）。

## Decisions

### 決定1: タイルコンポーネント（`WorkTile`, `PerformerTile`）の `Link` 化
タイル全体を `<Link>` コンポーネントに変更します。
* **理由:** HTML5 ではブロック要素を `<a>` で囲むことが許容されており、タイルのどの部分をクリックしても詳細画面へ遷移し、かつ全体がリンクとして中クリック等に対応できるため。

### 決定2: テーブルコンポーネント（`WorkTable`, `PerformerTable`）での `Link` 配置
`<tr>` 要素全体を `<Link>` で囲むことは HTML 仕様上不可能なため、テーブル内の「タイトル」および「出演者名」のセル内のテキストを `<Link>` コンポーネントに変更します。
* **理由:** 標準的な HTML マークアップに準拠しつつ、主要なアクセスポイントをリンク化するため。また、ホバー時に下線を表示するなどして、リンクであることがユーザーに伝わりやすくします。
* **代替案:** `onClick` で修飾キーを判定する（アプローチB）も検討されましたが、ブラウザにリンクとして認識されず、ホバー時にURLが左下に表示されない、右クリックメニューに「新しいタブで開く」が出ないといった問題があるため、今回の `Link` を使うアプローチAを採用しました。

## Risks / Trade-offs

- **[リスク]** タイルの `div` を `Link` に変更した際、Tailwind CSS やコンポーネント内のレイアウトが崩れる可能性がある。
  - **[対策]** `Link` に `display: flex` などの元の `div` が持っていたクラスをそのまま適用し、スタイルが変化しないか慎重に検証する。
- **[リスク]** テーブルの行全体をクリックして遷移する操作感が、テキストリンクのみのクリックになることで一部損なわれる。
  - **[対策]** 主要な情報（作品タイトル、出演者名）を大きくリンク化し、ホバー時のスタイルを明確にすることで、ユーザーの操作ミスを防ぐ。
