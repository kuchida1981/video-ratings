## Context

出演者一覧ページ（`PerformersPage.tsx`）は現在、バックエンドから取得した順（ふりがな→名前の固定ソート）でタイルを表示するのみ。ソートの切り替えができない。

バックエンドの `GET /performers` レスポンスには `work_count`（作品数）と `total_score`（出演者タグスコア合計）は含まれているが、各作品のスコア平均を表す `avg_work_score` は含まれていない。

## Goals / Non-Goals

**Goals:**
- 出演者一覧に「名前順 / 作品数順 / 作品平均点数順」× 昇順/降順のソートUIを追加する
- `avg_work_score` をバックエンドで計算し、`PerformerResponse` に追加する
- ソートの実行はフロントエンドのみで行う（APIを再コールしない）

**Non-Goals:**
- バックエンドへのソートクエリパラメータの追加
- ページネーション
- ソート状態の永続化（リロードでデフォルトに戻る）

## Decisions

### D1: ソートはフロントエンドのみで実行する

**決定**: APIを叩き直さず、取得済みの配列をフロント側でソートする。

**理由**: 現在ページネーションがなく全件取得済みのため、バックエンドにソートを委ねる必要がない。フロントのみであればUIの応答が即時になり、実装も単純。

**却下した選択肢**: クエリパラメータ（`?sort_by=name&sort_dir=asc`）をバックエンドに渡す案。ページネーション導入まで不要な複雑さを持ち込む。

---

### D2: `avg_work_score` はバックエンドで計算してレスポンスに含める

**決定**: `_build_performer_response` 内で各出演作品の `total_score` を集計し、平均を計算して返す。

**理由**: `work.total_score = 作品タグスコア + メイン出演者タグスコア` の計算には作品・タグの情報が必要で、フロントエンドは保持していない。フロントで計算するには追加APIコールが必要になり非効率。

**影響**: `list_performers` クエリに `joinedload` を追加して各作品の `work_tags` と `work_performers` をロードする必要がある。

**作品数が0のとき**: `avg_work_score = 0.0` とする。

---

### D3: 名前ソートはふりがな優先、フォールバックは name

**決定**: `furigana or name` の値で比較する。ふりがな未登録の場合は名前で代替。

---

### D4: UIコントロールの配置

**決定**: ページヘッダー行（「出演者一覧」の見出しと「新規登録」ボタンの間）にソートコントロールを配置する。ソートキーはセレクト、昇順/降順はトグルボタン。

## Risks / Trade-offs

- **クエリ複雑化**: `list_performers` のジョインロードが深くなる（WorkPerformer → Work → work_tags/work_performers）。出演者・作品数が増えると初回ロードが重くなる可能性がある → 現状の利用規模（個人利用）では許容範囲。将来的にはキャッシュや遅延ロードを検討。
- **avg_work_score の精度**: `calculate_work_total_score` はメイン出演者のスコアを含む計算であり、同一作品に複数出演者がいる場合でもメイン出演者のスコアのみを加算する設計。この挙動は既存ロジックを変えない。
