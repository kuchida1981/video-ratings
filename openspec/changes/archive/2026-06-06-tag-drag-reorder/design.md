## Context

タグ管理UI（TagsPage）では、タグカテゴリとその配下のタグが現状 DB の挿入順（id 昇順）で表示されている。`TagCategory` と `Tag` モデルには並び順を表すカラムが存在しないため、ユーザーが意図した順序に並べることができない。作品・出演者の詳細ページでのタグ付けUIも同じ API を参照しているため、タグ管理UIでの並び順がそのまま反映されるべきである。

フロントエンドはドラッグ&ドロップライブラリを現時点では持っていない。

## Goals / Non-Goals

**Goals:**
- `tag_categories` と `tags` テーブルに `sort_order` カラムを追加して並び順を永続化する
- タグ管理UIでドラッグ操作によるカテゴリ・タグの並び替えを実現する
- 並び替え結果が作品・出演者詳細ページのタグ一覧に自動的に反映される
- 新規作成されたカテゴリ・タグは末尾に追加される

**Non-Goals:**
- カテゴリを entity_type 間（work ↔ performer）で移動する機能
- 作品・出演者詳細ページでの個別タグ並び替え
- タグ付け一覧画面でのソート変更

## Decisions

### 1. `sort_order` カラムを DB に持つ

**決定**: `TagCategory.sort_order` と `Tag.sort_order` を `INTEGER NOT NULL DEFAULT 0` で追加する。

**理由**: アプリケーション外からのデータアクセスや API の直接呼び出しでも一貫した順序が得られる。フロントエンドでのみ管理する方式（JSON 配列を別テーブルに持つ等）と比べて、シンプルで既存クエリに `ORDER BY sort_order` を追加するだけで対応できる。

**代替案**: JSONB 列でカテゴリ・タグの順序配列を別に持つ → 正規化が崩れるため却下。

---

### 2. reorder API は `{ids: number[]}` のシンプルな配列を受け取る

**決定**: `PUT /tag-categories/reorder` と `PUT /tags/reorder` は `{ids: [3, 1, 2]}` 形式の body を受け取り、サーバー側で `sort_order = 0, 1, 2, ...` に採番する。

**理由**: クライアントは「最終的な順序」だけを知っていれば十分であり、sort_order の実値をクライアントが計算・管理する必要がない。dense な採番（0, 1, 2...）でよく、ドラッグ終了ごとに全件を送るため sparse 採番（0, 100, 200...）のメリットがない。

**代替案**: `[{id, sort_order}]` 形式 → クライアントに余分な責任を持たせるため却下。

---

### 3. ドラッグライブラリは `@dnd-kit` を採用

**決定**: `@dnd-kit/core` + `@dnd-kit/sortable` + `@dnd-kit/utilities` を追加する。

**理由**: React 向けのモダンなドラッグライブラリのデファクト。`react-beautiful-dnd` は非推奨であり、`@hello-pangea/dnd`（コミュニティフォーク）より公式メンテナンスが安定している。アクセシビリティ対応が充実しており、ライブラリの侵襲性が低い。

---

### 4. タグ管理UIのタグ表示を縦リスト形式に変更

**決定**: TagsPage のタグ表示をピル（`flex-wrap`）形式から縦リスト形式に変更し、各行にドラッグハンドル（`GripVertical`）を配置する。

**理由**: ドラッグ操作は縦リストの方が視覚的に自然で操作性が高い。作品・出演者詳細ページは変更なしのままピル表示を維持するため、管理UIと閲覧UIで表示形式が異なることになるが、目的が違うため問題ない（管理 vs. 付与操作）。

---

### 5. カテゴリのドラッグ範囲は entity_type セクション内に限定

**決定**: TagsPage は「作品用カテゴリ」と「出演者用カテゴリ」に分かれた2つの独立した sortable リストとして実装する。

**理由**: entity_type はカテゴリ作成後に変更不可（既存 spec）であり、セクションをまたいだ移動は意味を持たない。

---

### 6. 楽観的更新（optimistic update）を採用

**決定**: ドラッグ終了時にローカル state を即座に並び替えてから API を非同期呼び出しし、エラー時は `reload()` でサーバー状態に戻す。

**理由**: UI のレスポンスを高く保つため。reorder は冪等な操作であり、競合が発生しにくい（管理者1人が操作する想定）。

---

### 7. 新規作成時の sort_order は自動採番

**決定**: カテゴリ・タグ作成時、バックエンドが `MAX(sort_order) + 1` を自動的に割り当てる（フロントから sort_order を送らない）。

**理由**: 作成 API のシグネチャを変えずに済む。新規作成されたアイテムは常に末尾に追加され、その後ドラッグで移動できる。

## Risks / Trade-offs

- **マイグレーションのデータ初期化**: 既存データの `sort_order` は `id` 順で採番する（`ROW_NUMBER() OVER (ORDER BY id)`）。これにより既存の表示順が変わらない。→ マイグレーション SQL で対応。
- **並行操作**: 複数ブラウザ/セッションで同時に並び替えた場合、後勝ちになる。管理UIの用途（1ユーザーが手動で設定）に対しては許容範囲。
- **@dnd-kit の bundle サイズ**: 合計 ~25KB (gzip)。小規模アプリに対しては許容範囲。
- **タグ管理UIの見た目が変わる**: タグをピルから縦リストに変更するため、既存ユーザーには視覚的な変化がある。ただし管理目的のページなので機能的には同等以上。

## Migration Plan

1. DB マイグレーション `004_tag_sort_order.py` を追加
   - `tag_categories.sort_order` カラム追加（`DEFAULT 0`）
   - `tags.sort_order` カラム追加（`DEFAULT 0`）
   - 既存データの `sort_order` を `id` 昇順で採番
2. バックエンドのモデル・スキーマ・ルーターを更新
3. フロントエンドの依存インストール・型定義・TagsPage 実装
4. 動作確認後デプロイ（ロールバックは `sort_order` カラムを DROP するマイグレーション）
