## Context

対象デバイスは iPad Air 13インチ（論理解像度 ~1024×1366pt）。変更はフロントエンドのみで、バックエンド・API への変更はない。既存の smb-video-streaming 機能はバックエンドストリーミングが完成しており、フロントエンドの再生 UI のみ改善対象。

## Goals / Non-Goals

**Goals:**
- 作品詳細ページでの動画再生を1タップで開始・即全画面化する
- タイルグリッドの初期列数を iPad 画面幅に合わせる
- 作品一覧の絞り込みタグバッジ・Input を 44pt 以上のタップターゲットにする

**Non-Goals:**
- スマートフォン（~430px 以下）対応
- 編集フォーム・ボタン類のタッチ対応（タブレットでの編集は対象外）
- ナビゲーション構造の変更（サイドバーはそのまま）
- バックエンド変更

## Decisions

### 決定1: カバー画像エリアを再生ランチャーとして使用する

**採用**: カバー画像の `aspect-video` コンテナ上に smb:// ファイルの ▶ ボタンリストをオーバーレイ表示する。

**却下した案**: Files セクションの既存 play ボタンを拡大するだけ。
→ ファイルセクションがページ下部に埋もれており、再生意図を持ったユーザーが最初に目にするカバー画像エリアのほうが自然なエントリーポイントである。

**複数ファイル時**: カバー画像を常にオーバーレイで置き換え、ファイルリストを縦に並べて表示する（1ファイル時は ▶ 1つのみ）。

### 決定2: タップで即全画面 (`webkitEnterFullscreen`)

**採用**: ▶ タップ時に `video.webkitEnterFullscreen()` を呼び出してプログラム的に全画面を起動する。

**採用理由**: ユーザーが「再生後すぐ全画面にする」フローが多い。ネイティブの全画面ボタンを手動で押す手順を省く。

**フォールバック**: `webkitEnterFullscreen` が未定義の場合は `requestFullscreen()` を試みる（Chrome on Android 等）。

**video 要素の配置**: `<video>` は `sr-only` クラスで不可視のまま DOM 上に常駐し、`.play()` と `.webkitEnterFullscreen()` を `ref` 経由で呼び出す。視覚的な video 要素をページに表示する必要がないため、既存レイアウトを崩さない。

### 決定3: `window.innerWidth` によるレスポンシブデフォルト列数

**採用**: `localStorage` に保存値がない初回のみ `window.innerWidth` から列数を決定する（< 900px → 3、< 1200px → 4、それ以上 → 6）。ユーザーが設定変更した後は `localStorage` の値を優先する。

**却下した案**: CSS の `auto-fill` のみに任せる。
→ 現状の `minmax(100px, 1fr)` は maxCols=6 時に iPad portrait（~784px コンテンツ幅）で 130px/タイルになりタイトルが読みにくい。

### 決定4: 絞り込みバッジは CSS 調整のみ

**採用**: `WorksPage.tsx` の絞り込みバッジに `py-1.5` を追加、Input に `h-11` を付与する。

**却下した案**: Badge コンポーネント自体を変更する。
→ 詳細ページの編集用バッジまで影響が及ぶ。絞り込み用途のみに限定するため呼び出し側で追加する。

## Risks / Trade-offs

- `webkitEnterFullscreen` は iOS/iPadOS Safari 専用 API。PC Chrome 等でテストする場合は即全画面が動作しないが、ターゲットは iPad のため許容する → フォールバックで `requestFullscreen()` を呼ぶ
- Files セクションの既存 play/stop ボタンはそのまま残す。カバー画像エリアとの二重管理になるが、フォールバック UI として機能するため削除しない
- `window.innerWidth` は初期レンダリング時点の値を参照。デバイス回転後に再計算しないが、ユーザーが手動で列数を変更できる設定が既存にあるため許容する
