## Context

現在の CLAUDE.md では、agy を `agy --dangerously-skip-permissions --print "<プロンプト>" 2>&1` のワンショット形式で呼び出している。この方式には以下の問題がある:

1. **ハング**: agy が対話的入力（y/n プロンプト等）を求めるコマンドを実行し、応答待ちで停止する
2. **リカバリ不能**: タイムアウト機構がなく、ハング時に Claude Code が介入する手段がない
3. **一括委譲のリスク**: 大きなタスクを一度に渡すため、途中で失敗すると全体をやり直す必要がある
4. **コンテキスト消費**: Claude Code が `/code-review` で diff を全読みし、クレジットを消費する

agy は `--print-timeout`（デフォルト 5 分）と `--continue`（会話再開）を既にサポートしており、これらを活用する。

## Goals / Non-Goals

**Goals:**
- agy のハングを予防し、発生時にも自動復帰できるようにする
- タスク単位の分割呼び出しで、失敗の影響範囲を最小化する
- Claude Code のコンテキスト消費を削減する（agy に変更要約を生成させる）
- CLAUDE.md の記述を更新し、新しいパターンを標準ワークフローにする

**Non-Goals:**
- tmux / expect ベースのインタラクティブ制御（将来検討）
- agy の `-i`（インタラクティブモード）の導入
- OpenSpec のスキルやツール自体の変更
- agy のバージョンアップや設定変更

## Decisions

### D1: agy 呼び出しコマンドの標準形を変更する

**決定**: `--print-timeout` を必須パラメータとし、環境変数でハング予防する

```bash
GIT_TERMINAL_PROMPT=0 \
CI=true \
agy --dangerously-skip-permissions --print-timeout 3m --print "<プロンプト>" 2>&1
```

**理由**: agy にはデフォルト 5 分のタイムアウトがあるが、明示的に設定することで意図を明確にし、タスクサイズに応じた調整を可能にする。`GIT_TERMINAL_PROMPT=0` は git の認証プロンプトを抑制、`CI=true` は多くのツールの対話プロンプトを抑制する。

**代替案**: `timeout` コマンドでラップする → macOS に `timeout` が標準搭載されていない。agy 自身の `--print-timeout` の方がポータブル。

### D2: タスク単位の分割呼び出し

**決定**: OpenSpec tasks.md の各タスクを個別の agy ワンショットで実行する

```
従来:  agy 1回 → 全タスク実装
変更後: agy N回 → タスク1つずつ実装
```

**理由**: 
- 1タスクが失敗しても他に影響しない
- タスクごとに Claude Code がチェックポイントを入れられる
- タイムアウトを短く設定できる（小さいタスク = 短い実行時間）

**タイムアウトの目安**:
- 小タスク（1-2ファイル）: `--print-timeout 3m`
- 中タスク（3-5ファイル）: `--print-timeout 5m`
- 大タスク（6ファイル以上）: `--print-timeout 8m`（大タスクはさらに分割を検討）

### D3: タイムアウト・中断時のリカバリに `--continue` を使う

**決定**: タイムアウトした場合、`--continue --print` で同じ会話を再開し、状況に応じて再試行またはフォールバックする

```bash
# タイムアウト後の再開
agy --dangerously-skip-permissions --print-timeout 3m --continue --print \
  "前回タイムアウトした。対話的入力を求めるコマンドは使わず続行しろ" 2>&1
```

**リカバリフロー**:
1. タイムアウト発生
2. `git status` / `git diff` で途中成果を確認
3. 成果があれば `--continue` で残り作業を指示
4. 成果がなければ別アプローチで再試行
5. 2回再試行しても失敗 → Claude Code が直接実装

**代替案**: 毎回新しいセッションで再試行 → 途中まで進んだ作業のコンテキストが失われる。`--continue` なら agy 側に文脈が残る。

### D4: プロンプトにハング予防ルールを必ず含める

**決定**: agy への全プロンプトに以下のルールを標準で含める

```
制約:
- 対話的入力（y/n, パスワード等）を求めるコマンドは絶対に実行しないこと
- 必ず非対話フラグ（--yes, -y, --no-input 等）を付けること
- git push, npm publish など外部サービスへの送信は行わないこと
- 対話的入力が必要な状況に遭遇したら、実行せず [QUESTION] で報告すること
```

**理由**: `--dangerously-skip-permissions` は agy 自身のツール承認をスキップするが、agy が実行するシェルコマンドの対話プロンプトは防げない。プロンプトレベルでの予防が必要。

### D5: 別セッションレビュー + 変更要約

**決定**: 実装完了後、別の agy セッションでレビューと変更要約を生成させる

```bash
# 実装セッションとは別の新規セッションでレビュー
agy --dangerously-skip-permissions --print-timeout 3m --print \
  "以下の diff をレビューし、REVIEW セクション形式で報告しろ:
   1. 変更ファイル一覧と変更概要
   2. 潜在的な問題点（最大10個）
   3. 設計判断で迷った点
   4. テストの充足度
   $(git diff HEAD~N)" 2>&1
```

**理由**: 同一セッションの自己レビューは実装時のバイアスが残る。別セッション（フレッシュなコンテキスト）で実行することで「初見のレビュアー」として機能する。

Claude Code は agy のレビュー出力をもとに最終判断する。diff 全体を読む必要がなく、コンテキスト消費を抑制できる。

### D6: CLAUDE.md の標準フローを改訂する

**決定**: 「機能追加・バグ修正の標準フロー」のステップ3（実装）を分割呼び出しに、ステップ4（レビュー）を二段階に変更する

```
従来:
  3. 実装 (agy 一括)
  4. レビュー (Claude Code)

変更後:
  3. 実装 (agy タスク単位 × N)
  3.5. agy レビュー (別セッション)
  4. Claude 最終レビュー (要約ベース)
```

## Risks / Trade-offs

**[agy の `--continue` が期待通り動かない可能性]** → タイムアウト後の会話状態が壊れている場合は、新規セッションでフォールバックする。CLAUDE.md にフォールバック手順を明記。

**[タスク分割で agy 呼び出し回数が増える]** → Gemini 側のクレジット消費は増える可能性がある。ただし Gemini は利用量に余裕があり、Claude のクレジット節約が優先。小さいタスクは結合して呼び出す柔軟性を残す。

**[別セッションレビューの品質が不十分な可能性]** → agy のレビュー結果はあくまで参考情報。Claude Code が最終判断者であることは変わらない。レビュー品質が低ければ、このステップは省略可能とする。

**[環境変数 `CI=true` の副作用]** → 一部のツールが CI 環境と判断して動作を変える可能性がある。問題が発生した場合は環境変数を個別に調整する。
