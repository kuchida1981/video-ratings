## Context

作品詳細ページにはインライン動画再生機能がある。作品タグの評価は同ページで完結するが、出演者タグの評価には出演者詳細ページへの遷移が必要で、視聴しながらの一括評価ができない。

バックエンドの `_load_work()` はすでに `performer_tags` を `joinedload` しているため、DBクエリの追加は不要。レスポンス構造の拡張と、フロントエンドのUI変更が主な作業となる。

## Goals / Non-Goals

**Goals:**
- 作品詳細ページで出演者タグをインライン編集できる
- 出演者ごとの合計スコアを作品ページで確認できる

**Non-Goals:**
- 出演者評価を「この作品限定」にすること（評価は引き続きグローバル）
- 出演者タグ追加・削除用の新規APIエンドポイント作成（既存の `/performers/{id}/tags` を流用）
- 出演者カスタムフィールドの作品ページ内編集

## Decisions

### 1. 既存の出演者タグAPIを流用する

フロントエンドから `POST /performers/{id}/tags` / `DELETE /performers/{id}/tags/{tag_id}` を直接呼ぶ。作品ページ専用のAPIは作らない。

**理由**: 出演者タグはグローバルな評価であり、呼び出しコンテキストが変わっても操作は同一。エンドポイントを二重に持つ必要がない。

### 2. PerformerInWork スキーマを拡張する

作品取得レスポンス (`WorkResponse`) の `performers` フィールドに `tags` と `total_score` を追加する。

**理由**: `_load_work()` はすでに `performer_tags` を joinedload 済みのため、追加クエリなしで実現できる。フロントエンドは作品取得1回のAPIコールで全情報を得られる。

### 3. アコーディオン不使用・常時展開

出演者セクションで各出演者のタグUIを常に展開して表示する（折りたたみなし）。

**理由**: 出演者1人のケースが最多。1人ならアコーディオンは操作コストが増えるだけ。タグカテゴリ4つ・合計20個未満であれば常時展開でも視認性は許容範囲内。複数出演者のケースはまれなので縦積みで十分。

## Risks / Trade-offs

- [作品レスポンスのサイズ増加] 出演者タグ情報が加わるが、タグ数は少なく（20個未満）影響は軽微 → 許容
- [出演者ページとの二重編集] 両ページから同じタグを操作できるが、データは一元管理されているため整合性は問題なし → 許容
