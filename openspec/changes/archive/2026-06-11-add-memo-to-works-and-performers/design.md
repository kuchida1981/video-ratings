## Context

`Work` と `Performer` にはすでに `custom_fields: JSONB` カラムがあり、ユーザー定義の追加項目を格納できる。しかしこの仕組みは「設定ページでカスタム項目定義を作成してから使う」フローになっており、常に存在すべきメモ欄には適していない。また現在 `text` 型フィールドは `<Input>`（1行）でレンダリングされており、長文入力には不向きである。

## Goals / Non-Goals

**Goals:**
- 作品・出演者それぞれにフリーテキストメモを入力・保存できるようにする
- 詳細ページ下部（カスタム項目の下）に常に表示された Textarea として実装する
- ブラー時に自動保存し、ユーザーが保存ボタンを押す必要をなくす

**Non-Goals:**
- 一覧タイルへのメモ存在表示（アイコン含む）
- メモの全文検索
- メモのバージョン履歴
- カスタム項目システムの変更

## Decisions

### 1. `custom_fields` JSONB ではなく専用カラムを使う

**採用**: `memo TEXT` カラムを `works` / `performers` テーブルに追加する

**理由**: メモは設定不要で常に利用可能なファーストクラス機能であり、カスタム項目として扱うと「定義を作成してから使う」という不要なステップが生じる。また専用カラムにすることで将来的に全文検索インデックスや文字数カウントなどを追加しやすい。

**却下した案**: `custom_fields["memo"]` に格納する方法は設定不要だが、UIが単一行 Input になる・他のカスタム項目と区別がつかない・発見しにくい、という欠点がある。

### 2. ブラー時自動保存

**採用**: `<Textarea>` の `onBlur` ハンドラで既存の PATCH エンドポイントを呼び出す

**理由**: 「常に編集可能」というUX要件に最も自然にマッチする。保存ボタンが不要でページ上のノイズが減る。既存の PATCH `/works/{id}` / `PATCH /performers/{id}` が部分更新対応済みのため、`{ memo: value }` を渡すだけで実装できる。

**考慮点**: ブラー時保存では「変更したが保存されていない状態」がほぼ発生しないため、保存状態表示は省略する。

### 3. API 変更は最小限

`WorkUpdate` / `PerformerUpdate` スキーマに `memo: str | None = None` を追加するだけで、既存の PATCH エンドポイントがそのまま利用できる。新しいエンドポイントは不要。

## Risks / Trade-offs

- **ブラーが発生しないケース**: ページを閉じる・ブラウザをクローズするとブラーが発生しないことがある。→ 軽微な UX 欠点として許容する（`beforeunload` によるガードは Over-engineering）
- **同時編集の競合**: 複数タブで同一エンティティを開いて編集した場合、後からブラーが発生したほうで上書きされる。→ シングルユーザーアプリのため許容する

## Migration Plan

1. Alembic マイグレーションを生成・適用（`works.memo`, `performers.memo` 追加）
2. バックエンドのモデル・スキーマ変更
3. フロントエンドの UI 変更
4. ロールバック: マイグレーションを `downgrade` する（データ損失あり）
