## Context

`ImportPage.tsx` は単一コンポーネントで CSV インポートの全フローを担っている。現状の状態は `preview | null` と `result | null` の2変数で管理しており、フェーズの概念がない。テーブルは `max-w-3xl`（約 768px）内に5列を並べており、パスとエラー列が列見出しの時点で折り返す。インポートボタンは `is_valid === true` の全行を無条件に送信し、行ごとの選択・確認の手段がない。

## Goals / Non-Goals

**Goals:**
- テーブルを3列（チェックボックス・タイトル・出演者）に絞り、パス・エラーを副行で表示して折り返しを解消する
- 行ごとの選択トグルと全選択/全解除ボタンを提供する
- プレビュー → 確認 → 結果 の3ステップ遷移を実装する
- 確認画面では選択済みの valid 行のみを表示し、ここでも除外操作を可能にする

**Non-Goals:**
- バックエンド API の変更
- 仮想スクロール（DOM 上の行数制限）
- モバイル最適化

## Decisions

### 1. レイアウト: 2段行テーブルを採用

**選択肢:**
- A. 横スクロール付きテーブル（列幅確保）
- B. 2段行テーブル（パス・エラーを副行に）
- C. カードレイアウト

**決定: B**

カード（C）は100件超で縦スクロール量が多くなりスキャンしにくい。横スクロール（A）はパスとタイトル・出演者を同時に確認しにくい。2段行（B）は列見出しを3本に絞り、パスは `colSpan` で全幅に展開するため折り返しが起きにくく、かつ行の密度を保てる。

### 2. 状態管理: phase 変数 + selectedRows Set を追加

```ts
type Phase = 'upload' | 'preview' | 'confirm' | 'result'
const [phase, setPhase] = useState<Phase>('upload')
const [selectedRows, setSelectedRows] = useState<Set<number>>(new Set())
```

`preview` が `null` かどうかの判定から明示的な `phase` フラグへ移行することで、条件分岐が明確になる。`selectedRows` は `row_number`（サーバーが返す一意値）でキーイングし、`preview` データと独立して管理する。

**初期化タイミング:** プレビュー取得成功時に `is_valid === true` の `row_number` を全選択状態で Set を構築する。

### 3. エラー行の扱い: プレビューには表示・確認画面には非表示

エラー行はプレビュー上でグレーアウト + チェックボックス disabled として表示する（行の内容とエラー理由を確認できる）。確認画面には表示しない（インポートされないため不要）。

### 4. 全選択/全解除の対象: valid 行のみ

「全選択」「全解除」は `is_valid === true` の行のみを対象とする。エラー行は常に選択不可なので操作対象に含めない。

## Risks / Trade-offs

- [row_number の信頼性] サーバーが返す `row_number` がセッション内で一意であることを前提にしている → API 仕様上 CSV の行番号を返す設計のため問題なし
- [確認画面での追加操作不可] 確認画面で「選択済みでない行を追加したい」場合は「戻る」で前画面に戻る必要がある → UX 上許容範囲。確認画面は除外操作に特化する設計が明快
